# 实验目的

这个实验的目的，是为了验证：

1. `pca9548@0x73` 这颗 I2C mux 的 **channel 选通功能是否正常**
2. 在**绕过 Linux child bus 抽象**后，parent bus `i2c-0` 是否能直接访问指定 channel 下游的 PSU 设备
3. 不同 channel 下游是否真的独立，还是最终落到了**同一组底层数据源**
4. 问题到底更偏：
   - Linux child bus 访问逻辑
   - pca9548 选通逻辑
   - 还是 mux 下游物理通路/硬件隔离问题

------

# 实验原理

## 1. 系统结构背景

当前平台的 I2C 访问路径大致是：

```
AST2600 i2c-0
   ↓
pca9548 @ 0x73
   ↓
CH0 / CH1 / CH2 / CH3 / CH4 / CH5 / CH6 / CH7
   ↓
各 channel 下游 PSU FRU / PSU Ctrl 设备
```

Linux 为了方便访问，会把 pca9548 的每个 channel 抽象成一条 **child bus**，例如：

- `i2c-16`
- `i2c-20`
- `i2c-21`

但 child bus 访问会经过 Linux I2C core 和 `pca954x` 驱动自动选路，所以当访问结果异常时，不容易区分问题出在：

- child bus 逻辑
- pca9548 本体
- 还是下游硬件通路

------

## 2. 本实验的核心思路

本实验通过 **绕过 child bus**，直接在 parent bus `i2c-0` 上操作 `pca9548@0x73`，手工完成三件事：

### 第一步：手工选择 channel

直接写 pca9548 控制寄存器，指定当前接通哪一路 channel。

### 第二步：验证选择是否保持住

再读回 `0x73` 的寄存器值，确认 mux 是否真的停在对应 channel。

### 第三步：在 parent bus `i2c-0` 上直接访问下游设备

如果 channel 真的通了，那么 `i2c-0` 就应该能透过 mux 直接看到并访问对应 channel 下游的 PSU 设备，例如 `0x58 / 0x59`。

------

## 3. pca9548 控制寄存器原理

`pca9548` 是 8 通道 I2C mux，它的控制寄存器是一个 **bitmask**。

每一位对应一个 channel：

- bit0 → CH0
- bit1 → CH1
- bit2 → CH2
- bit3 → CH3
- bit4 → CH4
- bit5 → CH5
- bit6 → CH6
- bit7 → CH7

所以：

- `0x01` = `0000 0001` → 选通 **CH0**
- `0x10` = `0001 0000` → 选通 **CH4**
- `0x20` = `0010 0000` → 选通 **CH5**

------

# 测试过程

本实验按以下顺序进行。

------

## 步骤 0：读取 mux 初始状态

### 命令

```
i2cget -f -y 0 0x73
```

### 作用

读取 `pca9548@0x73` 的控制寄存器，确认实验开始前 mux 当前处于什么状态。

### 意义

- 如果读到 `0x00`，说明当前没有保持在某个 channel 上，处于空闲/断开状态。
- 如果读到非零值，说明某个 channel 当前正被选通。

------

## 步骤 1：手工选通 CH0

### 命令

```
i2cset -f -y 0 0x73 0x01
i2cget -f -y 0 0x73
i2cdetect -y -a 0
i2ctransfer -f -y 0 w1@0x58 0x9e r32
i2ctransfer -f -y 0 w1@0x59 0x9e r32
```

### 作用

- `i2cset`：命令 mux 选通 CH0
- `i2cget`：检查 CH0 是否保持住
- `i2cdetect -y -a 0`：看 parent bus 是否能透过 CH0 看到下游设备地址
- `i2ctransfer`：直接从 `i2c-0` 读取 CH0 下游 PSU 的字符串信息

### 实验含义

如果 CH0 选通成功且下游通路正常，则应看到：

- `0x73` 读回 `0x01`
- parent bus 上能扫到 `0x50 / 0x51 / 0x58 / 0x59`
- `0x58 / 0x59` 能直接读到下游数据

------

## 步骤 2：手工选通 CH4

### 命令

```
i2cset -f -y 0 0x73 0x10
i2cget -f -y 0 0x73
i2cdetect -y -a 0
i2ctransfer -f -y 0 w1@0x58 0x9e r32
i2ctransfer -f -y 0 w1@0x59 0x9e r32
```

### 作用

与 CH0 的验证完全相同，只是目标 channel 改为 CH4。

### 实验含义

如果 CH4 选通成功，则 `0x73` 应读回 `0x10`，并且 parent bus 可以透过 CH4 看到/访问下游设备。

------

## 步骤 3：手工选通 CH5

### 命令

```
i2cset -f -y 0 0x73 0x20
i2cget -f -y 0 0x73
i2cdetect -y -a 0
i2ctransfer -f -y 0 w1@0x58 0x9e r32
i2ctransfer -f -y 0 w1@0x59 0x9e r32
```

### 作用

与前两组一样，这次验证 CH5。

### 实验含义

如果 CH5 正常，`0x73` 应读回 `0x20`，并且 parent bus 可以直接读到 CH5 下游设备。

------

## 步骤 4：实验结束后再读一次 mux 状态

### 命令

```
i2cget -f -y 0 0x73
```

### 作用

确认实验完成后 mux 是否又回到空闲态。

### 意义

如果读到 `0x00`，通常说明：

- 当前没有保持在某个 channel
- 或系统配置了 idle-disconnect，实验后恢复到空闲状态

------

# 用到的命令与解析

------

## 1. `i2cget`

### 示例

```
i2cget -f -y 0 0x73
```

### 含义拆解

- `i2cget`：从 I2C 设备读取一个值
- `-f`：强制访问，即使内核认为该设备可能已被驱动占用
- `-y`：跳过交互确认
- `0`：使用 `i2c-0` 这条总线
- `0x73`：目标设备地址，这里是 pca9548 mux

### 在本实验中的作用

读取 mux 控制寄存器，确认当前 channel 选择状态。

------

## 2. `i2cset`

### 示例

```
i2cset -f -y 0 0x73 0x20
```

### 含义拆解

- `i2cset`：向 I2C 设备写入值
- `-f`：强制访问
- `-y`：跳过确认
- `0`：parent bus `i2c-0`
- `0x73`：mux 地址
- `0x20`：要写入的控制寄存器值，这里表示选通 CH5

### 在本实验中的作用

直接控制 pca9548 选通指定 channel。

------

## 3. `i2cdetect`

### 示例

```
i2cdetect -y -a 0
```

### 含义拆解

- `i2cdetect`：扫描某条 I2C 总线上的应答地址
- `-y`：跳过确认
- `-a`：扫描更完整的地址范围
- `0`：parent bus `i2c-0`

### 在本实验中的作用

在手工选通 channel 后，观察 parent bus 是否能透过该 channel 看到下游设备地址。

### 判读意义

- 能看到 `0x50 / 0x51 / 0x58 / 0x59`：该 channel 后面存在并应答这些设备
- 看不到：当前通路未建立，或下游设备不可达

------

## 4. `i2ctransfer`

### 示例

```
i2ctransfer -f -y 0 w1@0x58 0x9e r32
```

### 含义拆解

- `i2ctransfer`：执行更底层、更灵活的 I2C 事务
- `-f`：强制访问
- `-y`：跳过确认
- `0`：仍然使用 parent bus `i2c-0`
- `w1@0x58 0x9e`：先向地址 `0x58` 写 1 个字节 `0x9e`
- `r32`：然后从同一设备读 32 个字节

### 在本实验中的作用

直接读取下游 PSU 设备字符串寄存器，验证不只是地址存在，而是数据路径也真的可达。

### 判读意义

- 读成功：说明该 channel 下游设备真实可访问
- 读失败：说明该通路不可达，或设备未应答

------

# 结果判读原则

## 情况 A：写入值能保持，`i2cdetect` 能看到地址，`i2ctransfer` 也能读到数据

说明：

- mux 选通动作有效
- 对应 channel 下游物理通路可达
- 下游设备不仅地址存在，而且数据访问正常

------

## 情况 B：写入值能保持，但只能看到地址，读不到数据

说明：

- 通道表面上可见
- 但数据路径不完整，或目标设备响应异常

------

## 情况 C：写入值保持不住，读回 `0x00`

说明：

- 该 channel 没有稳定停留在选通状态
- 可能被切回空闲态，或该通路存在异常

------

## 情况 D：不同 channel 选通后，读到相同的字符串/序列号

说明：

- 虽然 Linux 和 mux 逻辑上认为是不同 channel
- 但这些 channel 后面最终落到了同一个底层数据源
- 更像 mux 下游物理通路共享或未隔离
